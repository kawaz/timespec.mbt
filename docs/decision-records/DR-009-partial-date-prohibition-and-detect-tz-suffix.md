# DR-009: 部分日付の禁止と detect_tz_suffix の新設

- **日付**: 2026-03-17
- **ステータス**: Accepted
- **変更元**: DR-008 セクション 3（部分日付対応）を破棄

## 問題

2つの問題が相互に関連していた。

### 問題 1: プラガブル parse_datetime での TZ 情報喪失

`parse_timespec` の Phase 4 で `parse_datetime~` を呼び出す際、シグネチャが `(String) -> Int64?` であるため、カスタムパーサが epoch ms のみを返し、パース時に検出した TZ 情報が失われる。

内部パーサ `parse_iso8601` は `parse_iso8601_with_tz` として `(Int64, TzOffset?)` を返す拡張版を持つが、Phase 4 ではカスタムパーサと内部パーサの両方を同じインターフェースで扱う必要があった。結果として、内部パーサ使用時も `parse_iso8601_with_tz` を二重呼び出しするか、TZ 情報を捨てるかという不自然な構造になっていた。

### 問題 2: 部分日付と短い TZ オフセットの曖昧さ

DR-008 セクション 3 で `YYYY`（年のみ）や `YYYY-MM`（年月のみ）を受け入れる決定をしていた。しかし `+/-HH` 形式の短い TZ オフセットとの曖昧さが判明:

- `2026-03-15T12:00:00+9` — `+9` は TZ オフセット（`+09:00`）
- `2026-03` — `-03` は月？TZ オフセット（`-03:00`）？
- `2026-01` — `-01` は月？TZ オフセット（`-01:00`）？

文字列末尾から TZ を検出する `detect_tz_suffix` を実装する上で、部分日付の存在がパターン判定を複雑かつ脆弱にしていた。

## 解決策

### 1. detect_tz_suffix の新設

`detect_tz_suffix(String) -> TzOffset?` を追加。文字列末尾から以下のパターンで TZ サフィックスを検出する:

| パターン | 例 | マッチ条件 |
|---|---|---|
| Pattern 1: `Z` / `z` | `...Z` | 末尾が `Z`/`z` |
| Pattern 2: `±HH:MM` | `...+09:00` | 末尾6文字 |
| Pattern 3: `±HHMM` | `...+0900` | 末尾5文字 |
| Pattern 4: `±HH` / `±H` | `...+09`, `...+9` | **符号の前が digit** かつ **文字列中にコロンあり** |

Pattern 4 の安全条件により:
- `2024-01-15` → `-15` の前は `0`（digit）だが、コロンなし → None（TZ 検出しない）
- `2024-01-15T12:00:00+9` → `+9` の前は `0`（digit）、コロンあり → `Hour(9)`
- `2024-01-15T12:00:00-5` → `-5` の前は `0`（digit）、コロンあり → `Hour(-5)`

この関数はパーサ非依存であり、`parse_datetime~` にカスタムパーサを渡した場合でも、パース成功後に元の文字列から TZ 情報を回収できる。これにより `parse_iso8601_with_tz` の二重呼び出しが不要になった。

### 2. 部分日付の禁止（YMD 必須）

`parse_iso8601` が年のみ（`2026`）や年月のみ（`2026-03`）を拒否するように変更。日付入力には YMD（年月日）すべてが必須。

これにより `detect_tz_suffix` の Pattern 4 が安全に機能する前提が成立する:
- `2026-03` のような入力は来ないため、`-03` を TZ と誤認するリスクがない
- 日付が完全形式（`YYYY-MM-DD`）の場合、末尾は必ず `-DD`（2桁の日）であり、`-H`（1桁の TZ）とは桁数が異なる

## 選択理由

### 部分日付禁止のトレードオフ

- **損失**: `--since 2026-03` のような省略入力が不可。`--since 2026-03-01` と書く必要がある
- **利得**: TZ オフセット検出の安全性が大幅に向上。パーサ非依存の TZ 検出が可能に

CLI 用途では「今月から」のような部分日付指定は稀であり、日付を数日分省略するよりも duration（`--since 30d`）で表現するのが自然。実用上の損失は小さいと判断。

### detect_tz_suffix vs parse_datetime シグネチャ拡張

代替案として `parse_datetime~` のシグネチャを `(String) -> (Int64, TzOffset?)?` に変更する方法も考えられたが:

1. **既存カスタムパーサの互換性が壊れる**
2. **カスタムパーサの実装者に TZ 検出の責務を課す**（本来パーサの関心外）
3. **detect_tz_suffix はパーサの種類に関わらず動作する**（ISO 8601 以外の形式でも TZ サフィックスが標準的な記法なら検出可能）

パーサ非依存のアプローチを選択することで、シグネチャの互換性を維持しつつ TZ 情報を回収できる。

## 影響

- `parse_iso8601`: 年のみ・年月のみの入力をエラーに変更
- `detect_tz_suffix`: 新規追加（内部関数）
- `parse_timespec` Phase 4: `detect_tz_suffix` を使用して TZ 情報を回収（`parse_iso8601_with_tz` の二重呼び出しを解消）
- DR-008 セクション 3: 本 DR により破棄（部分日付の受け入れ → 禁止に変更）
- DR-008 セクション 5: `@2026-01` の例が datetime ではなくエラーに変更
