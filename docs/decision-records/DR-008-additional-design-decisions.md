# DR-008: 追加の設計決定

- **日付**: 2026-03-16
- **ステータス**: Accepted

## 1. `@` マーカーの冪等性と datetime 制約

### 決定

- `@` は「この TimeSpec を Absolute として扱う」という冪等なマーカー
- `@` が複数あっても OK（`@@5m` = `@5m`、`1h@30m@` = `@1h30m`）
- **datetime リテラルは1パートに最大1つ**。2つ以上はパースエラー
- `@` + datetime は OK（`@` は冗長だが矛盾しない。datetime があれば暗黙 Absolute）

### 理由

`@` は基準点（now）の選択ではなく、出力形式の指定（ISO 8601 vs duration 文字列）。datetime リテラルは具体的な基準点を提供するので、2つあると基準が曖昧になるためエラー。`@` + datetime は「datetime が基準で、結果は Absolute」という意味で矛盾しない。

## 2. TzOffset の構造的 Eq と意味的 equal_offset

### 決定

- `TzOffset` の `derive(Eq)` は構造的等価のまま維持（`Hour(9) != Min(540)`）
- 意味的等価比較は `TzOffset::equal_offset` メソッドで提供（分換算して比較）
- `Eq` を意味的比較にオーバーライドしない

### 理由

意味的 Eq にすると `Local` の解決で FFI 副作用が `==` に入る。`==` が純粋でなくなるのは型の契約として不適切。構造的 Eq はパターンマッチとの整合性も保たれる。意味的比較が必要な場面（TzOffset 整合性チェック等）は `equal_offset` を使う。

## 3. ISO 8601 の部分日付・時刻のみ対応

### 決定

`parse_iso8601` が以下の省略形式を受け付ける:

| 入力 | 解釈 |
|---|---|
| `2026` | 2026-01-01T00:00:00（年のみ、月日は 01 デフォルト） |
| `2026-03` / `202603` | 2026-03-01T00:00:00（年月、日は 01 デフォルト） |
| `2026-03-01T12` | 12:00:00（分秒省略） |
| `2026-03-01T12:30` | 12:30:00（秒省略） |
| `10:30` | 1970-01-01T10:30:00（時刻のみ、日付は epoch） |
| `10:30:45.500Z` | 1970-01-01T10:30:45.500Z |

### 理由

ISO 8601 は部分表現を定義しており、曖昧さがない（4桁=年、HH:MM=時刻）。CLI 用途では `--since 2026-03` のように省略形で入力する場面が多い。time-of-day（`@10:30`）の実現にも time-only パースが必要。

## 4. デフォルト TZ は UTC（ISO 規格との意図的な乖離）

### 決定

TZ 情報のない datetime（`2026-03-15T12:00:00`）はデフォルトで **UTC** として解釈する。

### 理由

ISO 8601 では TZ なし = ローカル時刻が正式だが、ライブラリのデフォルトとして Local を採用すると:

1. **再現性が壊れる**: `Absolute` の「誰がいつ実行しても同じ結果」が保証できない
2. **環境依存のバグ**: 開発機（JST）と CI（UTC）で異なる結果になる
3. **WASM で Local が UTC フォールバック**: WASI に TZ API がないため、ターゲットで挙動が変わる

UTC デフォルトは安全側。アプリが Local を使いたければ `default_tz_offset~=Local` を明示するか、`make_parse_datetime(default_tz_offset=Local)` でパーサを注入する。

Design rationale: ISO 規格より実用上の安全性を優先。Go の `time.Parse` も同じ方針（デフォルト UTC、Local なら `ParseInLocation` を使う）。

## 5. `@DIGITS` で raw epoch ms（年ではない理由）

### 決定

`@` 付きで残り文字列が `[+-]?[0-9]+`（数字のみ、区切り文字なし）の場合、raw epoch ms として解釈する。datetime としては解釈しない。

- `@1704110400000` → `EpochTime(1704110400000)`
- `@2026` → `EpochTime(2026)` （2ms 後。年 2026 ではない）
- `@-100` → `EpochTime(-100)` （epoch の 100ms 前）
- `@2026-01` → datetime（区切り `-` があるので raw epoch ではない）

### 理由

1. **`date -d @EPOCH` との互換**: Unix の慣行で `@` + 数字 = epoch 値
2. **曖昧さ回避**: `@2026` を年と解釈すると `@2026` = `@2026-01-01T00:00:00Z` となるが、4桁の数字が年なのか epoch ms なのか判別できない。区切り文字（`-`/`/`）の有無で判定する方が明確
3. **実用性**: 大きな epoch 値（`@1704110400000`）は明らかに年ではない。小さな値（`@2026`）も一貫して epoch ms として扱う方が規則がシンプル
4. **年が必要なら区切りを書く**: `@2026-01` や `@2026-01-01` と書けば datetime として解釈される
